# 安全政策

## 支持版本

当前安全维护范围仅包括最新的 `main` 和当前 0.1.x MVP。项目不承诺长期支持周期、固定响应时间或生产级安全认证。

## 私密报告漏洞

优先使用托管仓库的 GitHub Private Vulnerability Reporting（如果项目所有者已启用）。如果该功能尚未启用，请创建一个**不含漏洞细节和敏感数据**的普通 Issue，请求维护者建立私密联系渠道。

不要在公开 Issue、Pull Request、日志附件或截图中提交：

- 真实聊天记录、Evidence 或 Profile
- API Key、访问令牌或 `.env`
- 真实 SQLite 数据库或上传文件
- 本机绝对路径或身份信息
- 完整漏洞利用代码、可直接利用的细节或未脱敏日志

请使用人工合成的最小复现，并只提供定位问题所需的信息。

## 安全范围

以下问题属于优先安全范围：

- 聊天、Evidence、Insight、Revision 或 Profile 数据泄露
- 路径遍历、任意文件读取或任意代码执行
- Origin/CORS 绕过或未授权本地 API 写入
- API Key、Authorization、环境变量或数据库 URL 泄露
- 未经服务端启用或逐请求授权就向远程 Provider 发送内容
- Profile 导出、临时上传文件或浏览器缓存造成的泄露
- 数据库破坏、证据链篡改或删除约束绕过
- ProfileSnapshot/InsightRevision 不可变性绕过
- 依赖供应链和发布产物完整性问题

一般 UI 问题、功能建议和不涉及敏感数据的文档错误请使用普通 Issue。

## 报告内容

在私密渠道中提供受影响版本、合成复现步骤、预期与实际行为、影响范围和已尝试的缓解方式。不要发送真实密钥或数据；如确有必要讨论敏感证据，请先与维护者确认安全传输方式。

维护者会在合理时间内确认报告；修复优先级和时间取决于严重性、可复现性和维护能力。
